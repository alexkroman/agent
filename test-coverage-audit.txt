ISSUE #1: packages/aai/_ssrf.ts -- COMPLETELY UNTESTED (Security-Critical!)
================================================================================

Oh! OH! This file implements SSRF protection for the ENTIRE self-hosted SDK
and it has ZERO dedicated tests! Yes, the builtin-tools.test.ts exercises it
indirectly through visit_webpage and fetch_json, but the core functions
`assertPublicUrl()`, `isPrivateIpv4()`, `ipToInt()`, and `ssrfSafeFetch()`

The server-side equivalent (_net.ts in aai-server) HAS its own test file.
The SDK-side copy does not. This is a security parity gap.

Recommended tests:
  - assertPublicUrl: test ALL private ranges (0.0.0.0/8, 10.x, 100.64.x,
    127.x, 169.254.x, 172.16-31.x, 192.0.0.x, 192.168.x, 198.18.x, 224.x,
    240.x) individually
  - assertPublicUrl: IPv4-mapped IPv6 bypass (::ffff:127.0.0.1)
  - assertPublicUrl: IPv4-mapped IPv6 hex format (::ffff:7f00:1)
  - assertPublicUrl: blocked hostnames (localhost, ::1, ::, *.local, *.internal,
    metadata.google.internal, 169.254.169.254)
  - assertPublicUrl: allowed public IPs (8.8.8.8, 1.1.1.1)
  - assertPublicUrl: blocked protocols (ftp:, file:, javascript:)
  - assertPublicUrl: allowed protocols (http:, https:)
  - ssrfSafeFetch: redirect chain with all-safe URLs succeeds
  - ssrfSafeFetch: redirect to private IP is blocked mid-chain
  - ssrfSafeFetch: exceeding MAX_REDIRECTS (5) throws "Too many redirects"
  - ssrfSafeFetch: redirect with no Location header returns response
  - ipToInt: edge cases (0.0.0.0, 255.255.255.255)
